## Critical directories explained

### `scripts/`
The **only directory with imperative, testable code** at the repo level. All three scripts are stand-alone bash. install.sh has the largest blast radius (touches the user's `.claude/`, `.cursor/`, `_bmad/`, and every git repo's `.git/hooks/`). Test priority: install.sh > adoption-dashboard.sh > check-skill-symlinks.sh.

### `hooks/prepare-commit-msg`
Single file, 31 LOC. The hook that fires on every commit in any repo where the installer has run. Logic surface: source detection (skip merge/squash), trailer pre-existence check, branch-name parsing for `Story-Ref`. Bug here = corrupt commit messages everywhere.

### `templates/`
Static files copied verbatim into target workspaces. No logic. Test surface = "did install.sh copy them correctly?", which collapses into install.sh tests.

### `claude/skills/` and `cursor/skills/`
**These mirror each other.** Confirmed via spot-check; only `.DS_Store` differs. Each skill follows one of three structural patterns:

| Pattern | Files | Examples |
|---|---|---|
| **Steps-driven** | SKILL.md, workflow.md, steps/ or steps-c/ subfolder with N step files | `bmad-create-prd` (12 steps), `bmad-code-review` (6 steps) |
| **XML-driven** | SKILL.md, workflow.md (XML tags inline, no steps subfolder) | `bmad-dev-story` (~23KB XML workflow) |
| **Rule-only** | SKILL.md only — no workflow, no steps | `dontbmad-caveman` |

A few skills additionally bundle scripts (`bmad-distillator` has Python helpers + pytest) or templates.

## Entry points

| What | Where |
|---|---|
| User runs installer | `scripts/install.sh` |
| User views dashboard | `scripts/adoption-dashboard.sh` (after install copies it into target workspace) |
| Git fires on commit | `<repo>/.git/hooks/prepare-commit-msg` (copied from `hooks/`) |
| Agent invokes skill | `~/.claude/skills/<name>/SKILL.md` or `<workspace>/.claude/skills/<name>/SKILL.md` |
| Agent loads rule at session start | `<workspace>/.cursor/rules/*.md` or `<workspace>/.claude/rules/*.md` |

## What's NOT in this tree

- No `package.json`, `pyproject.toml`, `Makefile`, `justfile`, or any build system at the repo level.
- No `.github/workflows/`, no GitLab CI, no CircleCI, no CI/CD of any kind.
- No `tests/` directory at the repo level — the only test file lives inside one skill.
- No language runtimes (no `node_modules`, no `.venv`).

This is a **pure-distribution repo** — content + bash glue. The test-suite PRD addresses what happens when CI is added.
